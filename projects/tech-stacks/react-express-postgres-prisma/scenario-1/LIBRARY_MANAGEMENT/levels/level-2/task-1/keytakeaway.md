# Key Takeaway

Creating reusable utility functions is the foundation of clean, maintainable full-stack code. When you write the same logic in multiple places, any change requires updating every instance—increasing the chance of bugs and making your code harder to maintain. In a React + Express + Prisma application, creating a dedicated availability helper ensures consistent logic across your frontend and backend.

**Why this is important:**

Pure functions with predictable outputs are easier to test and debug, especially when working with Prisma queries and React components. A pure function always returns the same output for the same input and doesn't modify anything outside itself. This predictability is crucial because when something goes wrong in your code, you can immediately identify whether the issue lies in your input data or somewhere in a hidden side effect. Without pure functions, debugging becomes a nightmare of tracing through mutable state and understanding which function changed what. When you're querying PostgreSQL through Prisma and passing data to React components, pure functions ensure your data transformations work correctly every time.

Centralizing business logic means changes only need to be made in one place, which is essential for maintaining large applications. Imagine you have availability checking logic spread across your Express routes, React components, and maybe even some Prisma middleware—a book list page, a search results page, a book detail page, a borrowing form, and an admin dashboard. Now imagine business requirements change: perhaps the library is undergoing renovation and all books should be unavailable during construction hours, or perhaps there's a new policy that reference books cannot be borrowed. Without a centralized function, you'd need to update five different places, carefully ensuring each modification is consistent. With a centralized function used by both your Express API and React frontend, you update one place and all components automatically get the correct behavior.

Reusable utilities can be imported anywhere they're needed, reducing code duplication significantly across your full-stack application. When you create a well-named, focused function that does one thing well, both your Express backend and React frontend can use it. This creates a cohesive codebase where the same reliable logic is used everywhere, rather than having slightly different versions of similar logic scattered throughout the project. With Prisma, you might create a utility that checks availability by querying the database once and caches the result, which can then be used by multiple API endpoints.

When business rules change, you only need to update one function instead of many, which makes your code adaptable to changing requirements. Business requirements will always change—that's the nature of software development. New regulations might require different availability rules, seasonal events might affect borrowing periods, or new book categories might have special rules. Having centralized logic in your Express backend means these changes are straightforward. You modify one function, run your tests, and you're done.

This principle of creating reusable functions extends far beyond utilities—any repetitive code should be refactored into reusable functions, components, or modules. It's one of the most important skills for writing maintainable software with React, Express, PostgreSQL, and Prisma that can grow and evolve over time.
